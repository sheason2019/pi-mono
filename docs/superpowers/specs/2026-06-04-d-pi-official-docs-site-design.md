# d-pi 官方文档站 — Design

**Date**: 2026-06-04
**Status**: Approved (brainstorming complete)
**Package scope**: `packages/d-pi-official/` (new sub-package).
No modifications to `packages/coding-agent/` or any other existing package.

## Context

d-pi 的功能已经基本收口（远程执行器、多 agent 树、sources、auth 等都落地了），
但目前没有任何对外文档——README 不写、CHANGELOG 是 release notes 不是 user guide、
设计文档存在 `docs/superpowers/specs/` 里但那是给开发者看的。用户上手 d-pi
只能靠 `d-pi --help` 摸索。

需要一个面向终端用户的文档站，让"已经决定用 d-pi 的人"和"正在评估的人"
能在 5 分钟内跑起来、按功能查到命令/工具用法。

## Goal

用 Docusaurus 搭一个 d-pi 官方文档站，覆盖 d-pi 全部已实现功能，
每功能页用「一句话 → 用法 → 参数 → 示例 → 相关」统一模板，配内联 bash 示例。

## Audience

**主**：d-pi 终端用户（评估 / 上手 / 日常查命令 / 查工具参数）。
**次**：d-pi 内部开发者（设计文档在 `docs/superpowers/specs/`，不进站点）。
**不服务**：上游 pi 用户（他们看 `packages/coding-agent/docs/`，不进 d-pi 站点）。

## Non-Goal

- 部署到公网（v1 只跑本地 `docusaurus start` / `docusaurus build`）
- 多语言 i18n（v1 只有 `zh-CN`）
- 多版本并行（v1 只生成 v0.6 一份快照；v0.7 出来再切）
- Algolia 搜索（用本地搜索）
- 写 tutorial / walkthrough / 单独 examples 项目
- 引 UI 库 / 分析 / 评论 / 任何外部脚本

## Decisions (brainstormed)

| 维度 | 选择 | 理由 |
|---|---|---|
| Audience | 终端用户 | 当前最大价值是把功能讲清楚，不是暴露内部架构 |
| Content depth | 简洁参考（A） | 每功能 1 页 = 简介 + 用法 + 示例 + 相关，28 篇覆盖全 |
| Deployment | 本地 only（B） | 还在 alpha，URL/域名都没定，先铺内容 |
| IA | 按功能聚合（A） | 工具型产品按"我要做 X"找文档，按功能分块最易跳转 |
| Language | 纯中文（B） | 用户日常工作语言一致 |
| Examples | 内联代码块（A） | 简洁参考风格天然内嵌最自然 |
| Title | `D-Pi Agent Teams` | 用户指定 |
| Tagline | `轻松编排具有你个人特色的 Agent 团队` | 用户指定 |
| Logo | Placeholder（`static/img/logo.svg`） | 后续有真 logo 再换 |
| Versioning | 启用（B） | 当前内容快照为 v0.6 |
| Search | 本地搜索（A） | 零配置够用 |
| Locale | `zh-CN` 默认（A） | 干净 URL，UI 中文化 |
| 包位置 | `packages/d-pi-official/` | 用户指定 |
| Sidebar | 手写（不用 autogenerated） | feature-grouped 结构清晰稳定 |

## Tech Stack

- `docusaurus` v3.10.x（classic preset + TypeScript config）
- `@easyops-cn/docusaurus-search-local`（本地搜索，Docusaurus 3.x 兼容）
- `typescript` ^5.x（仅 devDep，用于 `docusaurus.config.ts` / `sidebars.ts` 类型）
- 跟随的 React 等 transitive 依赖由 Docusaurus 自带

**不引入**：`tailwindcss` / shadcn / GA / Plausible / Algolia / Disqus / 任何 UI 库 / 任何分析。

## Page Template

每篇功能页（**非** overview 页）统一 6 段结构：

```markdown
---
title: create_agent
sidebar_position: 2
---

# create_agent

一句话：在当前 agent 下创建一个子 agent，返回子 agent 的 id。

## 用法

工具名 `create_agent`。Agent 在需要把任务委派给子 agent 时调用。

## 参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `name` | string | 是 | 子 agent 名字，hub 内唯一 |
| `prompt` | string | 否 | 子 agent 的初始 system prompt |
| `parent_id` | string | 否 | 父 agent id；默认调用方自己 |

## 返回值

```json
{
  "id": "uuid-xxx",
  "name": "researcher",
  "parent_id": "uuid-yyy"
}
```

## 示例

**场景**：root agent 创建一个负责查文档的子 agent。

```bash
# Agent 在 LLM 决策时输出
create_agent(name="researcher", prompt="你是文档查询专家")
```

**预期返回**：
```json
{ "id": "a1b2-...", "name": "researcher", "parent_id": "<caller-id>" }
```

随后可用 `send_message(agent_id="researcher", message="...")` 派活。

## 相关

- [agent_network](./agent-network) — 查子 agent 是否已存在
- [send_message](./send-message) — 派活给子 agent
- [destroy_agent](./destroy-agent) — 收尾时清理

## 注意事项

- 子 agent 创建后会进入 `starting` 状态，等其首个 LLM 调用完成才进入 `running`
- `name` 一旦指定不可改；如需重命名，destroy + create
```

**模板变体**（按页面类型选用对应字段名）：
- **工具页**（`create-agent` / `send-message` / ...）：name / 参数 / 返回值 / 示例
- **CLI 页**（`install` / `init` / `serve` / `users` ...）：命令 / 标志 / 示例
- **Slash 命令页**（`/sources` / `/agents`）：命令 / 行为 / 示例

**「注意事项」**段按需出现（不是每页都有），只在有坑时写。

**Overview 页**（5 个 section 各自的 `overview.md`）**不用模板**，用：
1. 1-2 段说清这个能力领域解决什么问题
2. 子页面链接列表
3. 一个 5-10 行的端到端最小例子

**首页（`docs/intro.md`）**用 doc-style landing：
1. **d-pi 是什么**（一段话，说明跟 pi 的关系）
2. **d-pi 能解决什么问题**（多 agent 树形编排 + 远程工具执行 + 跨 agent 寻址）
3. **下一步**（"先看 [快速上手 → 安装](./getting-started/install) 5 分钟跑起来"）

## Scripts

```json
{
  "scripts": {
    "start": "docusaurus start --port 3000",
    "build": "docusaurus build",
    "serve": "docusaurus serve --port 3000",
    "clear": "docusaurus clear",
    "typecheck": "tsc --noEmit",
    "check": "npm run typecheck && npm run build"
  }
}
```

`check` 是 `docusaurus-official` 子包内部的兜底——`tsc` + 完整 build，含 broken-link 检查。
**不接入 monorepo 根 `npm run check` 链**——d-pi-official 是独立子项目，跟 pi-mono
的 build/test 解耦。

## Build & Dev Workflow

**首次启动**：

```bash
cd packages/d-pi-official
npm install
npm run start         # http://localhost:3000
```

**日常开发**：
- 改 `docs/**/*.md` → 浏览器自动 reload
- 改 `docusaurus.config.ts` / `sidebars.ts` / `src/css/*` → 需刷一次页面
- 改 `package.json` / 装新依赖 → `npm install` + 重启

**写完后落地**：
- `npm run build` 通过 + 浏览器内 spot-check 新页面
- 不引未在 `package.json` 里声明的依赖
- `git add packages/d-pi-official/docs/<新文件> ...` 显式列路径，不用 `-A`
- 改 `docs-site/**` 时不碰 `packages/d-pi/src/**`（除非同步要改功能文档里的代码示例）

## Versioning

v1 启用 versioning 但**只生成 v0.6 一份快照**：

```bash
# 在 packages/d-pi-official/ 跑一次，把当前 docs 快照为 v0.6
npm run docusaurus docs:version 0.6
```

之后 `docs/` 下的内容变成 `current`（开发中），`versioned_docs/version-0.6/` 是稳定的 v0.6 快照，
sidebar 顶部显示 `0.6 (current)` 下拉——v0.6 之前不会有别的选项。

后续打 v0.7 标签时再 `docs:version 0.7` 切第二份快照。

## Open Questions / Deferred to Implementation

实现阶段如有歧义，按以下优先级决定（避免反复打断）：

1. **代码示例中命令输出**——以仓库当前 main 分支实测为准，写完跑一次 `d-pi --help` 校对
2. **架构图**——用 SVG 占位框图（不上 Mermaid / PlantUML），保持 diff 友好
3. **首页 hero 风格**——用 Docusaurus 默认 doc landing，不用自定义 React 页面
4. **custom.css 范围**——只改 `code` 字体和 toc 颜色，不动主题色

## Out of Scope (v1)

明确**不**在 v1 范围内，避免 scope creep：

- 部署到任何公网 URL / 配 CI / 写 GitHub Actions
- 多语言 i18n / `i18n/en/` 目录
- 多版本并行（v0.5 / v0.4 ...）
- Algolia DocSearch 集成
- 写 tutorial / walkthrough / 单独 examples 项目
- 引 UI 库 / Tailwind / 分析 / 评论
- 引 Mermaid / PlantUML 插件（架构图用 SVG）
- 自定义 `src/pages/*` React 首页
- 从代码注释自动生成 API 参考
- 文档站本身的 doctest / mdx 组件化测试
- PDF 下载 / RSS / 搜索键盘快捷键
